# AIエージェント × ノーコード構想

**Agent-Centric Architecture Vision Document**

## 1. ビジョン

AIエージェントをシステムの中心に据え、ユーザーは「意志」を伝えるだけで、
プロジェクト管理・コンテンツ制作・SNS発信・開発・ナレッジ蓄積までを一気通貫で実行できる基盤を構築する。

### 1.1 コアコンセプト：

ユーザーが「こうしたい」という意志を持てば、実現手段の選定・ワークフローの構築・API連携・実行までをAIエージェントが自律的に行う。

- **従来のアプローチ:** ユーザーがツールを選び、設定し、操作する
- **本構想のアプローチ:** ユーザーは意志を伝え、エージェントがすべてを実行する

### 1.2 レイヤー構造：バイブコーディング × ノーコード

本構想の根幹にある設計思想は「バイブコーディングの上にノーコードを載せる」というレイヤー構造である。

| レイヤー | 役割 | 具体例 |
|---|---|---|
| 中心層（AIエージェント） | 意思決定・コード生成・API実行・ワークフロー構築 | Claude Code, Claude API |
| 外側（ノーコードUI） | ユーザーが触れるインターフェース。操作の「カバー」 | Notion共通基盤, Rajitan管理画面 |
| 入力チャネル | ユーザーの意志をエージェントに届ける経路 | 会話（テキスト/音声）、GUI操作、図示 |

この構造はn8n等の既存ノーコードツールとは逆のアプローチである。n8nはノーコード基盤が「下」にあるため、その設計思想や制約から逃れられない。本構想ではコード（バイブコーディング）が主導権を持ち、GUIはあくまで「触りやすくするためのカバー」に過ぎない。必要ならGUIを無視してコード側から何でも操作できる。

## 2. 既存アセット

本構想は、既に開発済みまたは設計済みの複数のシステムを統合する形で実現する。

### 2.1 共通基盤システム（Notion）

「木」のメタファーに基づくタスク管理兼プロジェクト管理基盤。Portal（テーマ別ダッシュボード）を起点に、成果物（Nuts）・イシュー（Trunk）・タスク（Leaf）・ナレッジ（Root）・リソース（Resource）を一元管理する。

- Portal → DB_Tag → 各DBの間接参照によるフィルタリング
- ゲーミフィケーション（経験値・レベル・称号）による継続動機
- アイゼンハワーマトリクスによる優先度の自動判定
- マイルストーンの自動分割とフェーズ管理

### 2.2 Rajitan（らじたん）

Discord会話体験向上Bot。会話の自動要約・クイズ生成・音楽推薦などを、ラジオDJ風に演出する。

- 会話監視と自動要約による知識の構造化
- キャラクター設定によるインタラクティブな対話
- Web管理画面（ダッシュボード・統計・設定）が実装済み
- LeveMagi（ゲーミフィケーション）機能を搭載

### 2.3 関連する技術資産

- Notion API連携の知見
- Claude API / Claude Codeの活用経験
- 画像生成AI（nanobanana等）の調査
- Discord Bot開発の実装経験
- バイブコーディング × ノーコードの設計思想

## 3. アーキテクチャ

### 3.1 全体構成

システムの中心にAIエージェントが位置し、周辺のすべてのツール・サービスをエージェントが統合的に操作する。

```
            ┌─────────────────────────┐
            │    ユーザーの意志        │
            └────────────┬────────────┘
                         │
            ┌────────────┴────────────┐
            │   AIエージェント（中心） │
            └─┬────┬────┬────┬────┬─┘
              │    │    │    │    │
        Notion  Discord 画像生成 SNS  カレンダー
        共通基盤 Rajitan  API   API    API
```

### 3.2 エージェントの責務

| 責務 | 詳細 |
|---|---|
| 意図解釈 | ユーザーの自然言語入力から、実行すべきアクションを判断する |
| ツール選択 | Notion / Discord / 画像生成 / SNS等、どのサービスを使うかを自動判断する |
| ワークフロー構築 | 複数ステップにまたがる処理を自律的に設計・実行する |
| コード生成 | 必要に応じてスクリプトやAPI呼び出しコードを生成・実行する |
| 提案 | ワークフローの改善案やタスクの優先順位を能動的に提案する |

### 3.3 入力チャネル

ユーザーはエージェントに対して複数の方法でアクセスできる。どの入力方法でも、最終的にエージェントが処理を実行する。

- テキスト会話（自然言語で指示）
- GUI操作（Notion共通基盤、Rajitan管理画面）
- Discord経由（Rajitanを通じた対話）
- 音声入力（録音 → テキスト変換 → エージェントに送信）

## 4. ユースケース

### 4.1 ジャーナリング → SNS自動発信（実証ケース）

本構想の最初の実証ケースとして、ジャーナリングからSNS発信までの自動化を想定する。

1. ユーザーがジャーナリングアプリまたは会話でテキストを入力する
2. エージェントがテキストを受け取り、DB_Root（ナレッジ）にシードとして保存する
3. エージェントがアカウントごとにシステムプロンプトを切り替え、文体をリライトする
4. 画像生成APIを叩き、アカウントに合ったデザインの画像を生成する
5. 各SNSアカウントに自動投稿する

**ポイント:** ユーザーは「書く」だけ。ワークフローの設計・実行・投稿はすべてエージェントが行う。同一人物バレの問題も、文体変換と画像デザインの差異で自動的に解消される。

### 4.2 プロジェクト管理の自動化

- 「このアイデアを形にしたい」→ エージェントがNutsを作成、タスク分解してLeafに登録
- 作業進捗に応じてフェーズ・アイゼンハワーマトリクスを自動更新
- 次にやるべきタスクをエージェントが能動的に提案

### 4.3 会話からのナレッジ蓄積

- RajitanがDiscord上の会話を自動要約
- 要約からアイデアやナレッジを抽出し、DB_Rootに自動登録
- シードとして蓄積されたナレッジを、エージェントが定期的にレビュー・昇格提案

### 4.4 開発支援

- 自然言語で機能要件を伝えると、エージェントがコードを生成
- 既存コードベースの理解・修正・テストもエージェントが実行
- API連携のコード生成（Notion API、Discord API、画像生成API等）

## 5. 競合との差別化

| 観点 | Google (Gemini + Workspace) | n8n等のノーコード | 本構想 |
|---|---|---|---|
| 中心にあるもの | ツール群 | ノーコード基盤 | AIエージェント |
| ユーザーの役割 | ツールを選んで操作 | ワークフローを設計 | 意志を伝えるだけ |
| カスタマイズ性 | Googleの枠内 | ノーコード基盤の制約内 | コードレベルで無制限 |
| UX思想 | ギーク寄り・高機能 | GUI操作前提 | 人間の不器用さを前提にした設計 |
| パーソナライズ | 汎用的 | 自分で設定 | 個人の思考体系に最適化 |
| 動機づけ | なし | なし | ゲーミフィケーション |

### 5.1 差別化の核心

Googleは「優秀なツール群」を提供し、n8nは「管理しやすい自動化基盤」を提供する。本構想は「意志さえあれば回る仕組み」を提供する。レイヤーが異なるため、Googleのサービスすら部品として組み込める立場にある。

また、Googleがやらない（できない）領域として、個人の思考体系に合わせた管理体系がある。共通基盤のゲーミフィケーション、ポータル構造、木のメタファーによる階層管理は、汎用ツールでは実現しない深いパーソナライズである。

GoogleがNotionの代替ができない（しない）ように、本構想はUXと人間中心設計で勝負する。管理能力が低い人でも、続けられる仕組みを提供することが最大の差別化ポイントである。

## 6. 実装ロードマップ

### Phase 0: 検証（現在）

- 最初の実証ケース（ジャーナリング → SNS自動発信）の最小構成を構築
- エージェントの意図解釈 → API実行 → 結果返却のパイプラインを検証
- 既存アセット（共通基盤・Rajitan）との接続ポイントを確認

### Phase 1: 最小構成の実装

**決定事項:**
- エージェントの核に何を使うか（Claude API直叩き / Claude Code / LangChain等）
- データの永続化先（Notion共通基盤をそのまま使うか、別DBを立てるか）

**実装内容:**
1. エージェントが会話を受け取り意図を解釈する層
2. 意図に応じてAPIを叩く実行層（Notion API / 画像生成API / SNS API）
3. 結果をユーザーに返すレスポンス層

### Phase 2: 統合と拡張

- Rajitanとの連携（Discord経由の入力チャネル）
- 共通基盤のタスク・ナレッジとの双方向連携
- ワークフローの自動提案機能
- 音声入力チャネルの追加

### Phase 3: ノーコードGUI層

- エージェントの操作状況を可視化するダッシュボード
- ワークフローの視覚的な編集インターフェース
- 非技術者向けの簡易操作UI

## 7. 未解決の論点

| 論点 | 検討事項 |
|---|---|
| エージェントの核技術 | Claude API直叩き vs Claude Code vs LangChain。自由度・コスト・保守性のバランス |
| データ永続化 | Notion APIの制約（レートリミット等）で問題ないか。専用DB（SQLite/Supabase等）が必要か |
| コスト構造 | AI API利用料、画像生成API利用料、ホスティング費用の見積もり |
| セキュリティ | ユーザーのSNSアカウント認証情報の安全な管理方法 |
| スケーラビリティ | 個人利用から他者提供への展開可能性。マルチユーザー対応の設計 |
| エージェントの自律性の範囲 | どこまで自動実行し、どこで人間の確認を挟むか。特にSNS投稿の承認フロー |
